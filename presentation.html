Le Moteur de Bases de Données (SGBD)

⚙️ Le moteur (SGBD) = le logiciel qui fait vivre la base de données : il stocke, lit, écrit et protège les données.

🚗 Métaphore

La base de données, c'est la voiture entière. Le moteur (SGBD), c'est ce qui la fait avancer : sans lui, les données sont juste des fichiers immobiles sur un disque.

🍕 Les distributions, c'est comme la pizza !

Une recette de base (la pâte = PostgreSQL), et chacun ajoute sa garniture : plus de sécurité, du support 24/7, une version cloud... Même cœur, saveurs différentes !

🐘 Les distributions PostgreSQL

PostgreSQL Community — la version originale, gratuite et open-source
EDB Postgres Advanced — version entreprise : sécurité et support renforcés
Amazon Aurora / RDS — version cloud gérée par Amazon (AWS)
Azure Database for PostgreSQL — version cloud gérée par Microsoft
Google AlloyDB / Cloud SQL — version cloud gérée par Google

Comment est Organisée une Base de Données ?

🏠 Métaphore : une BDD est comme une grande maison bien rangée — chaque élément a un rôle précis !

🏠 Schéma

Les étages de la maison : regroupent les tables par thème (ventes, RH...)

🗄️ Table

Les armoires : chaque table range un type d'information (élèves, notes...)

📋 Colonne

Les tiroirs étiquetés : nom, classe, note — chaque colonne a un type précis

🔖 Index

L'index d'un livre : trouver une info sans parcourir toutes les pages

🤖 Procédure

Le robot ménager : une suite d'actions enregistrée, exécutable à la demande

🧩 Extension

Les modules en plus : ajoutent des super-pouvoirs (cartes, recherche...)

🎯 Pourquoi cette structure ? Sans organisation, retrouver une donnée = chercher une chaussette dans une maison en désordre. Avec : tout est rangé, rapide et fiable !

Et Sous le Capot ? La BDD Côté Système

🍳 Métaphore : la BDD fonctionne comme une cuisine de restaurant — chaque zone a son rôle !

💽 Le disque (file system)

= Le garde-manger 🥫
Les données sont stockées dans des fichiers sur le disque : durables, mais lents d'accès. Tout y survit même quand on éteint le serveur.

⚡ La mémoire (RAM)

= Le plan de travail du cuisinier 🔪
Ultra-rapide mais limitée : on y garde les données « chaudes » utilisées souvent. Tout disparaît si on éteint !

🔄 Le va-et-vient permanent

Quand on demande une donnée : le moteur regarde d'abord en mémoire (le plan de travail). Si elle n'y est pas, il va la chercher sur le disque (le garde-manger) — plus lent ! Le travail du DBA : régler la mémoire pour que les données les plus demandées soient toujours « à portée de main ».

💡 RAM = 1000× plus rapide que le disque, mais 10× plus chère — tout l'art est de bien doser les deux !

SQL — Parler avec les Données

🗣️ Comment interagir avec une BDD ? On lui parle dans son langage : SQL (Structured Query Language).

💡 SQL est comme une question posée à un bibliothécaire : « Donne-moi tous les livres de science-fiction publiés après 2020. »

SELECT  Lire des données — « Montre-moi les notes d'Alice »
INSERT  Ajouter — « Ajoute le nouvel élève Karim »
UPDATE  Modifier — « Corrige la note de Bob : 15 »
DELETE  Supprimer — « Retire l'élève parti du lycée »
Exemple de requête SQLSELECT nom, note FROM eleves WHERE classe = '2nde A' ORDER BY note DESC;

Automatiser : l'Ancienne Méthode (Scripts)

Avant les outils modernes, les DBA écrivaient des scripts Shell (Bash) ou Python pour automatiser leurs tâches.

backup_db.sh — Script de sauvegarde#!/bin/bash # Sauvegarde PostgreSQL DATE=$(date +%Y%m%d) DB_NAME="production" pg_dump $DB_NAME \ > /backup/db_$DATE.sql echo "Backup terminé : $DATE"

✅ Avantages

▸ Simple à écrire rapidement
▸ Flexible, personnalisable
▸ Gratuit, aucun outil externe

❌ Inconvénients

▸ Fragile : casse si l'environnement change
▸ Pas idempotent (exécuter 2× = problème)
▸ Un script = un seul serveur à la fois
▸ Difficile à maintenir avec le temps

💡 Idempotent = si tu exécutes le script 1 fois ou 10 fois, le résultat est le même. Un script Bash dit « créer », mais si c'est déjà créé → ERREUR !

Les Scripts, en Images 🎭

📖 Le script = une recette de cuisine écrite à la main

Elle marche très bien... dans TA cuisine. Mais chez quelqu'un d'autre (four différent, ingrédients manquants), la recette échoue sans prévenir.
➜ Un script dépend de son environnement : changement de serveur = risque de panne.

🤖 Le script = un robot qui suit les ordres aveuglément

Tu lui dis « ajoute du sel » : il en ajoute, même si le plat est déjà salé ! Il ne vérifie jamais l'état avant d'agir.
➜ Exécuter 2 fois un script = doublons, erreurs... C'est le problème d'idempotence.

✉️ Le script = envoyer 100 lettres une par une

Pour 100 serveurs, il faut se connecter à chacun et lancer le script à la main, l'un après l'autre. Long et épuisant !
➜ Pas de gestion de parc : le script ne connaît qu'une machine à la fois.

Et si un outil savait vérifier avant d'agir, et parler à 1000 serveurs en même temps ? 🤔

Ansible — Automatiser en Décrivant l'État Voulu

🤖 Ansible est un outil open-source qui automatise la configuration de serveurs en décrivant l'état qu'on veut obtenir.

install_postgres.yml — Playbook Ansible- name: Installer PostgreSQL hosts: all # 100 serveurs d'un coup tasks: - name: Installer le paquet apt: name: postgresql state: present - name: Démarrer le service service: name: postgresql state: started
🔁

Idempotent

Exécutable 100 fois : même résultat. Aucune surprise !
🌐

Multi-serveurs

1 playbook configure 1 ou 1000 serveurs simultanément.
📖

Lisible

YAML = langage humain. L'équipe entière peut comprendre.
🔌

Sans agent

Rien à installer sur les serveurs cibles. Juste SSH.
Scripts Bash : "Exécute ces commandes"
Ansible : "PostgreSQL doit être installé et démarré" → Ansible vérifie et corrige si nécessaire

Ansible vs Script Shell — Cas Concrets

Cas d'usage📜 Avec un script Shell🤖 Avec Ansible
💾 Sauvegarder 50 BDD chaque nuit1 script par serveur, à copier et adapter 50 fois. Si un serveur change de mot de passe → le script casse en silence.1 playbook + 1 inventaire de 50 serveurs. Exécution parallèle, rapport clair : 48 OK, 2 en échec.
🏗️ Installer PostgreSQL sur 20 serveursSe connecter à chaque serveur un par un, lancer le script, vérifier à la main. Relancer = risque d'erreur « déjà installé ».1 commande pour les 20 serveurs. Relançable à volonté : Ansible vérifie l'état et ne refait que ce qui manque.
🔐 Appliquer un patch de sécurité urgentPlusieurs heures de connexions manuelles. Aucune trace de qui a patché quoi, ni quand.Patch déployé sur tout le parc en quelques minutes, avec un log complet et auditable de chaque action.

🏆 Bilan : moins d'erreurs, plus de vitesse, et tout est tracé — c'est ça, l'automatisation moderne !

Industrialisation — Produire à Grande Échelle

🏭 Analogie : une usine Toyota produit la même voiture, fiable, 1000 fois par jour. L'industrialisation, c'est ça !

📐 Standardisation

Tous les serveurs suivent les mêmes règles : mêmes configs, mêmes versions, mêmes noms.

🔁 Reproductibilité

Dev, Test, Production = environnements identiques. Fini le « ça marche chez moi ! »

📦 Infrastructure as Code

Les configs sont du code : versionnées dans Git, révisées, auditables.

🚀 Déploiement Continu

Une nouvelle BDD est toujours déployée de la même façon, en 5 minutes, pas en 5 jours.

🎯 Automatisation + Industrialisation = faire en 1 semaine ce qui prenait 3 mois à la main !
